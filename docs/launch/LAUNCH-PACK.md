# Launch Pack

Use this pack to launch Awesome Agent Workflows without fake engagement, spam, or hype. The goal is simple: help real developers discover a repo that saves them time when operating coding agents.

Primary link: https://github.com/Richicinschi/awesome-agent-workflows

## Positioning

One-line pitch:

> Copy-pasteable workflows for shipping real work with AI coding agents safely.

Expanded pitch:

> Awesome Agent Workflows is a vendor-neutral library of operating procedures for coding agents. Each card gives a trigger, scoped agent brief, verification gates, stop conditions, and final evidence format so people can use Claude Code, Codex CLI, Cursor agents, Aider, OpenCode, local agents, or custom assistants without trusting vibes.

Contrast:

- Not an agent framework.
- Not a prompt dump.
- Not a leaderboard.
- A practical runbook library for agent-assisted software work.

## Audience

Prioritize people with an immediate pain:

- developers using coding agents in real repositories;
- maintainers reviewing AI-generated pull requests;
- teams experimenting with parallel agents or worktrees;
- local-agent and personal-AI-OS builders who need safer operating loops;
- engineering leads trying to turn “agent usage” into repeatable team practice.

## Launch principles

- Ask for feedback before asking for stars.
- Lead with a concrete workflow example, not a generic repo announcement.
- Share only where project posts are welcome.
- Do not use star exchanges, paid stars, fake accounts, mass comments, or unrelated issue comments.
- Respond quickly and convert good feedback into issues, docs, or workflow cards.

## Example-led snippets

Fresh-context review:

> AI-generated PRs can look convincing even when verification was skipped. The fresh-context review workflow treats the agent summary as untrusted and checks the diff, tests, changed files, and evidence from a clean context before merge.

Parallel agents:

> Running three agents at once is easy. Integrating them without branch chaos is the hard part. The parallel-agent and worktree-lane workflows show how to split scope, isolate branches, and verify integration.

CI red-to-green:

> Do not rerun CI until it turns green and call that fixed. The CI red-to-green workflow forces a reproducer, explains environment deltas, and blocks weakening checks without approval.

Release safety:

> Agent-generated release work needs rollback thinking. The release runbook workflow makes the agent report readiness evidence, rollout steps, rollback triggers, and residual risks before launch.

Tool safety:

> Tool-using agents need permission boundaries. The MCP/tool integration review workflow maps tool access, data flow, secrets exposure, and human approval gates before trusting a new integration.

## Reply templates

If someone asks “how is this different from a prompt library?”:

> The workflow cards include prompts, but the main value is the operating contract around them: trigger, scope, non-goals, verification gates, stop conditions, and final evidence format. The goal is to make agent work auditable and repeatable, not just to get better one-off generations.

If someone says “agents still make mistakes”:

> Agreed. The repo assumes agents make mistakes. That is why the workflows emphasize fresh-context review, regression tests, verification evidence, and human approval gates for risky changes.

If someone asks for vendor-specific support:

> The core cards are vendor-neutral. Tool adapters are welcome when they explain exactly where to paste the brief, what permissions the tool has, and how to verify the output.

If someone reports a missing workflow:

> That is exactly the kind of contribution this repo needs. If you can describe the trigger, failure mode, and verification evidence, it can become a workflow card.

If someone asks for stars directly:

> If it helps you, starring makes it easier for other agent users to find. Feedback and missing workflow requests are even more valuable right now.
