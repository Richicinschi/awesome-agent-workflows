# Posting Checklist

Use this checklist to run a feedback-first launch for Awesome Agent Workflows.

Primary link: https://github.com/Richicinschi/awesome-agent-workflows
Launch copy: [Launch Pack](LAUNCH-PACK.md)
Growth policy: [Ethical Growth Plan](GROWTH-PLAN.md)
Issue seeds: [Community Issue Seeds](COMMUNITY-ISSUE-SEEDS.md)

## Preflight

- [ ] Repo is public.
- [ ] README top section explains the repo in one sentence.
- [ ] Demo GIF loads from the README.
- [ ] Topics include `ai-agents`, `coding-agents`, `awesome-list`, `claude-code`, `codex`, and `developer-tools`.
- [ ] Contribution guide and curation policy are linked.
- [ ] Labels exist for `workflow request`, `resource suggestion`, `good first issue`, `documentation`, and `help wanted`.
- [ ] At least three participation issues are open.
- [ ] The launch post asks for feedback, not only stars.
- [ ] Any channel-specific rules have been read before posting.

## Day 0: seed useful participation

- [ ] Open issue: workflow request for reviewing agent-generated tests.
- [ ] Open issue: tool adapter request for an additional coding agent.
- [ ] Open issue: good first issue for a real-world agent failure story.
- [ ] Open issue: resource suggestion for public examples of agent runbooks.
- [ ] Pin or bookmark the catalog, demo GIF, and contribution guide for fast replies.

## Day 1: close network launch

Start with people likely to give substantive feedback.

- [ ] Share in trusted chats or communities where agent workflows are relevant.
- [ ] Ask 5 real coding-agent users for feedback on missing workflows.
- [ ] Record each link and useful reply below.
- [ ] Convert good suggestions into issues within the same day.

## Day 2: public launch post

Pick one main public launch venue first. Avoid posting everywhere at once; it is easier to respond well when feedback is concentrated.

Recommended first public venue:

1. Show HN if the post can be framed as lessons learned and practical value.
2. X or Bluesky thread if the account has agent-builder followers.
3. Reddit only where project posts are allowed and the framing is community-specific.
4. LinkedIn or Mastodon for team/process audience.

After posting:

- [ ] Reply to factual questions with specific workflow links.
- [ ] Thank criticism and turn specific gaps into issues.
- [ ] Avoid arguing about generic AI hype.
- [ ] Do not ask people to star in every reply.
- [ ] Mention stars only when someone says the repo helped them.

## Day 3-4: convert feedback into repo improvements

- [ ] Add or improve one workflow based on feedback.
- [ ] Add a tool adapter if multiple users ask for the same agent.
- [ ] Add one worked example if users ask how the workflows look in practice.
- [ ] Keep README concise; put details in docs.
- [ ] Close or update seed issues that become real work.

## Day 5-7: second wave

Use a different angle from the first post.

Possible second-wave angles:

- “Fresh-context review catches agent mistakes before merge.”
- “How to run parallel coding agents without branch chaos.”
- “A CI red-to-green workflow for agent-generated changes.”
- “A public repo launch workflow that explicitly avoids fake stars and spam.”
- “Tool-permission review for MCP and tool-using agents.”

Before each second-wave post:

- [ ] Link one concrete workflow directly.
- [ ] Include the repo link only once.
- [ ] Ask a specific question.
- [ ] Confirm the post is allowed in that community.

## Success metrics

Track these as signals, not as vanity goals:

- people name a missing workflow;
- people open issues or PRs;
- stars increase after useful discussion;
- other repos link to a workflow card;
- someone reports using a workflow in a real PR, bug fix, release, or incident.

## Stop conditions

Stop or slow down if any of these happen:

- posts are being removed as off-topic;
- replies indicate the value proposition is unclear;
- feedback repeats the same missing context;
- issue replies cannot be handled promptly;
- the repo starts attracting low-quality promotional links.

## Follow-up response rhythm

During the first 48 hours:

- Check comments and issues every few hours while awake.
- Respond with workflow links and short explanations.
- Open issues for good suggestions.
- Merge small doc fixes quickly after verification.
- Save bigger feature requests for labeled issues.

## Suggested posting order

1. Private/close network feedback request.
2. One X/Bluesky thread or LinkedIn/Mastodon post.
3. Show HN when ready to monitor comments for several hours.
4. One relevant Reddit community, adapted to that community.
5. Second-wave example post using a specific workflow story.
